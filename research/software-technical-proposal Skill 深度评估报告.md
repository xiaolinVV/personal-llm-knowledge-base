# software-technical-proposal Skill 深度评估报告

## 总体判断

先给结论：这个 skill 的大方向是对的，但还没有达到“高频、稳定、低风险地生成正式交付级技术方案”的状态。它最正确的地方有四个：第一，强调“先读采购文件和评分办法，再写方案”；第二，把模板定义成“可裁剪骨架”而不是固定目录；第三，明确区分叙述型正式方案与矩阵式逐条响应；第四，把实施、质保、售后、培训、验收、质量控制纳入技术方案正文，而不是默认拆成孤立附件。你的这些设定，和公开采购文件中“商务技术文件”“采购需求偏离表”“合同条款偏离表”“技术/服务响应承诺及资料”并存的真实做法是一致的。fileciteturn0file0 citeturn14view0turn14view1turn16view1turn20view0turn23view0

但它现在也有几个真实问题，而且其中有些已经属于 **P0 级别**。最明显的不是“文风不够好”，而是“规则边界和模板边界还不够稳”。公开法规要求招标文件按标准文本编制，并且至少包含商务条件、采购需求、资格条件、报价要求、评标方法与标准、合同文本；对于需要供应商提供设计方案、解决方案或者组织方案的项目，评审因素要尽可能客观、量化，合同还要覆盖验收、交付、质保、知识产权、风险管控等内容。换句话说，**技术方案从来不是一篇纯散文，也不是一张纯答题卡**，而是“叙述性正文 + 必要附表/偏离表/承诺表”的组合体。你当前 skill 对“正文去过程化”的理解是对的，但对“法定格式”和“附表响应”的兼容性还不够强。citeturn23view0turn3view2turn22view1turn22view2

我认为目前最需要你直修的硬问题有五个。其一，**全局规则和 reference 模板存在内部冲突**：SKILL.md 要求输出目录使用 `生成文件/<template_id>/...` 隔离不同模板产物，但 `formal-procurement-tech-solution`、`compliance-matrix-procurement-response`、`fujian-soe-steady-narrative-tech-solution` 的 reference 里仍大量使用通用的 `生成文件/01-交付文档/`、`生成文件/02-图示源码/` 等路径，会直接导致同一项目多模板产物互相覆盖或混放。其二，**formal 与 fujian 两个叙述型模板都缺“法定附表/偏离表附录模式”**，这会让模型在真实招采场景里被迫在“像正式方案”和“满足格式要求”之间二选一。其三，**`fujian-soe-steady-narrative-tech-solution` 有明显过拟合**，如“20天节点安排”“三年质保”等项目特定表达，不适合作为泛化骨架。其四，**“默认多图”写成了过强的全局铁律**，这对架构型方案是加分项，但对逐条响应、运维服务、暗标/双盲场景并不总成立。其五，**暗标/双盲约束没有被提升为独立模板级规则**，这在当前各地去标识化评审和“双盲”推进背景下，风险偏高。fileciteturn0file0 fileciteturn0file1 fileciteturn0file2 fileciteturn0file3 citeturn26view1turn26view2turn9search0turn28search11

再明确一点：**我没有找到“全国统一的软件项目技术方案官方模板”**。我找到的是两类东西：一类是国家层面的采购法规，要求招标文件、合同、评审规则按标准文本和法定内容编制；另一类是地方财政或采购平台发布的公开招标/磋商示范文本，例如北京、福建等地的示范文本，它们都强调“按项目特点具体化填写”“可以补充或修改表格格式”，而不是提供一个全国统一的软件技术方案正文模板。因此，你的 skill 不应声称模板来自官方统一标准；它应明确表述为：**基于公开采购文件、地方示范文本、典型评分办法和行业交付惯例归纳出来的写作模板体系**。citeturn23view0turn15view0turn10search2turn10search5

我单独强调两个最重要的判断。

第一，`formal-procurement-tech-solution` 现在最像“通用母模板”，但它要避免当成“万用模板”。公开评分里，常见重点是 **项目理解、技术设计、接口/部署、实施方案、质量控制、培训验收、售后支持**；我看到比较多的是把“技术设计方案”细化为技术架构、数据结构、应用接口、网络架构、系统部署，而不是把“创新性”单独拉成很重的一章。你现在这个模板如果继续保留“创新性与适用性”默认成章，模型会很容易写成“AI 味”最重的一部分。更稳的办法是：**只有评分办法明确存在“创新性/先进性/可扩展性”时才单列；否则并入总体设计或“方案特点与适配性说明”**。fileciteturn0file2 citeturn27search1turn14view2turn19search15

第二，`fujian-soe-steady-narrative-tech-solution` 的问题不是文风不好，而是 **触发条件比模板自己声称的更窄**。公开资料里，我能找到的更接近它风格的，是集团管控、国资在线监管、数据采集交换、长期运维这类项目，它们确实更强调责任链、台账、监管对象、事前-事中-事后控制、持续优化；但我没有找到“福建国企风格”作为官方模板分类的规范依据。也就是说，这个模板可以保留，但你应把它理解成一个 **面向集团治理/监管/长期运营项目的稳健叙述模板**，而不是一个泛化到所有福建地区政企软件项目的默认风格。fileciteturn0file3 citeturn28search2turn27search0turn11view1
